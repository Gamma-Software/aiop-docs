# Sélection des déclarations

La sélection des déclaration est un mécanisme qui permet de filtrer les déclarations en fonction du système cible et de ses paramètres. Cela permet de définir des déclarations spécifiques à un système cible, et de les rendre disponibles uniquement pour ce système cible.

## Système cible

Les systèmes cibles compatibles avec le playbook est difini par le fichier `.aiop/inventory.yml`. L’utilisateur final indique ensuite le système cible pour lequel il souhaite générer le playbook. Le système cible sera pris en compte pour filtrer les déclarations.

### Exemple

Si nous reprenons l’[exemple de la société Tesla ](/docs/configurations/inventory#exemple) le système cible pourrais être:
- `suv/modelX/phase2`
- `sedan/model3/phase2`
- `truck/cybertruck/phase1`

Ou encore:
- `suv/modelX/phase1`
- `sedan/modelS/phase1`

<Callout type="warning" emoji="⚠️">
 Vous devez séparer les éléments de votre système cible par des slash `/` et respecter l’ordre définit dans le [fichier de configuration de l’inventaire](/docs/configurations/inventory#exemple-de-configuration).
</Callout>

## Compatibilité

Les déclarations sont également filtrées en fonction de la compatibilité avec le système cible. La compatibilité d'une déclaration est définie dans le champ `compatibility` de la déclaration.
Reprenons l’exemple de la société Tesla avec le système cible `truck/cybertruck/phase1`.

### Système compatible

Vous pouvez déclarer que la déclaration est compatible avec un système spécifique. Mais aussi avec une gamme de modèles ou avec tous les modèles d'une architecture. Dans ce cas, tous les models de la gamme `model_range1` de l’architecture `archi1` sont compatibles avec la déclaration. Ici nous déclarons le binaire d'un logiciel de monitoring de couple de remorquage compatible compatible avec le système `truck/cybertruck/phase1`:

```yaml {4}
- source: software/towing-torque-monitor/1.24.100/bin
  destination: software/towing-torque-monitor
  compatibility:
    - truck/cybertruck/phase1:
        functions: [towing]
        envs: [dev, test]
```

Similairement vous pouvez indiquer que la déclaration est compatible avec toutes les gammes de modèles Cybertruck (si une gamme de Cybertruck existait) ou de l'architecture `truck`:

```yaml {4}
- source: software/towing-torque-monitor/1.24.100/bin
  destination: software/towing-torque-monitor
  compatibility:
    - truck/cybertruck:
        functions: [towing]
        envs: [dev, test]
```

Enfin vous pouvez indiquer que la déclaration est compatible avec tous les systèmes avec la valeur `all`:

```yaml {4}
- source: software/towing-torque-monitor/1.24.100/bin
  destination: software/towing-torque-monitor
  compatibility:
    - all:
        functions: [towing]
        envs: [dev, test]
```

Vous constaterez que les champ `compatibility` attend un liste de combinaison système/paramètres compatibles. Il est donc possible de déclarer pour une ressource plusieurs combinaisons. Dans ce cas vous pouvez déclarer que la ressource est compatible avec 2 phases de développement spécifiquement (`phase1`et `phase2`):

```yaml {4-12}
- source: software/towing-torque-monitor/1.24.100/bin
  destination: software/towing-torque-monitor
  compatibility:
    - truck/cybertruck/phase1:
        functions: [towing]
        envs: [dev, test]
    - truck/cybertruck/phase2:
        functions: [towing]
        envs: [dev, test]
```

### Paramètres de compatibilité

Dans notre cas `functions` et `envs` sont les paramètres à remplir pour indiquer les fonctionalités et les environnements compatibles avec la déclaration. Concentrons nous sur le paramètre `envs` pour l'instant.

```yaml {6}
- source: software/towing-torque-monitor/1.24.100/bin
  destination: software/towing-torque-monitor
  compatibility:
    - truck/cybertruck/phase1:
        functions: [towing]
        envs: [dev, test]
```

Pour rappel les valeurs autorisée de `envs` pour le système `truck/cybertruck/phase1` sont `test`, `bench`, `prod` et `dev` et que la règle de sélection est `one_of` ([revoir l'inventaire](/docs/configurations/inventory#modèle-de-configuration)). Dans ce cas nous avons la possibilité de déclarer que la ressource est uniquement compatible avec l’environnement `test`:

```yaml {6}
- source: software/towing-torque-monitor/1.24.100/bin
  destination: software/towing-torque-monitor
  compatibility:
    - truck/cybertruck/phase1:
        functions: [towing]
        envs: [test]
```

Ou bien avec les environnements `test` et `bench`:

```yaml {6}
- source: software/towing-torque-monitor/1.24.100/bin
  destination: software/towing-torque-monitor
  compatibility:
    - truck/cybertruck/phase1:
        functions: [towing]
        envs: [test, bench]
```

Ou bien avec tous les environnements:

```yaml {6}
- source: software/towing-torque-monitor/1.24.100/bin
  destination: software/towing-torque-monitor
  compatibility:
    - truck/cybertruck/phase1:
        functions: [towing]
        envs: [test, bench, prod, dev]
```

Ou encore avec la valeur `all` pour indiquer que la ressource est compatible avec tous les valeurs possibles d'`envs`:

```yaml {6}
- source: software/towing-torque-monitor/1.24.100/bin
  destination: software/towing-torque-monitor
  compatibility:
    - truck/cybertruck/phase1:
        functions: [towing]
        envs: [all]
```

Il n'y a pas d'ordre dans la liste de valeurs d'`envs`. Cependant, il est recommandé de les organiser de manière logique par rapport aux autres déclaration pour faciliter la lecture et la maintenance.
